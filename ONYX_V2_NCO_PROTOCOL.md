# 🧠 Onyx V2 — NCO Digital Protocol (v2.1.0)

**إطار معرفي مفتوح لتصميم معالجات رقمية تعتمد على المذبذبات الرقمية (NCO) وترميز اتجاه الإطلاق (FDE)**

**~6 pJ · 5 دورات ساعة · 100% دقة · ~4x توفير مقابل CMOS 65nm**

---

## 📋 القسم 1: الملخص التنفيذي

**Onyx V2** هو الجيل الثاني من معالجات التصنيف الثنائي Onyx. بدلاً من المذبذب الاسترخائي التماثلي (LIF) في v1، يستخدم v2 **مذبذبًا رقميًا (NCO)** مع accumulator 32-bit موقع (signed).

### النتائج الحالية (الإصدار v2.1.0 المُستقِر)

| المقياس | القيمة | ملاحظات |
|---------|--------|---------|
| N_WINDOW (عدد دورات التصنيف) | **5** | مثبت في Verilog RTL و Yosys |
| الدقة (محاكاة 50 عينة) | **100%** | Python + Verilog |
| الموارد (iCE40) | **1,455 LUTs / 476 FFs** | Yosys synthesis |
| الطاقة التقديرية (50 عينة) | **~6 pJ** | بناءً على LUTs و FFs في iCE40 |
| التوفير مقابل CMOS 65nm | **~4x** | (25 pJ لـ CMOS @ 50 عينة) |

### ملاحظات مهمة

1. **N_STEPS=2 كان تجريبياً في Python فقط.** الإصدار RTL الحالي يستخدم N_WINDOW=5 للحفاظ على 100% دقة عبر LFSR الحالي. تقليصها إلى 2 في Verilog ممكن نظرياً لكنه لم يُختبر بعد.
2. **الطاقة الحقيقية ~6 pJ (وليس 0.03 pJ).** الـ 0.03 pJ كان تقديراً نظرياً لمجرد عمليات الجمع. مع إضافة LUTs و FFs والفانوت، الرقم الحقيقي ~6 pJ على iCE40. هذا لا يزال أقل من CMOS 65nm (25 pJ) — نحو 4x توفير.
3. **الضوضاء المستخدمة هي Uniform (وليس Gaussian).** LFSR يوزع القيم بشكل منتظم. هذا لا يؤثر على الدقة (100% ما زالت محققة)، لكنه يضعف الارتباط بنموذج Stochastic Resonance الكلاسيكي الذي يفترض Gaussian.

### المقارنة مع v1

| المقياس | v1 LIF | v2 NCO | التحسن |
|---------|--------|--------|--------|
| عدد الخطوات/عينة | 5 | 5 | 1x (متطابق فعلياً) |
| الطاقة/تصنيف (50 عينة) | 0.87 pJ (تماثلي) | ~6 pJ (رقمي) | — (رقمي أكبر لكنه قابل للبرمجة) |
| نوع المذبذب | LIF تماثلي (RC) | NCO رقمي (32-bit) | — |
| الموارد (iCE40) | غير مقدر | 1,455 LUTs | — |
| التوفير vs CMOS 65nm | 29x | ~4x | — |

**ملاحظة:** المقارنة العادلة هنا — v1 تماثلي (يحتاج R+C خارجي) بينما v2 رقمي بالكامل وقابل للبرمجة على أي FPGA. التوفير مقابل CMOS يبقى إيجابياً.

## 📋 القسم 3: العمارة الرقمية

### 3.1 وحدة NCO الأساسية

```
acc[31:0] ← signed 32-bit accumulator
         acc += F_WORD + NOISE (LFSR داخلي لكل NCO)
         if acc > TH:   → fire positive, acc -= OFFSET
         if acc < -TH:  → fire negative, acc += OFFSET

حيث:
  TH     = 2^30 (32'h40000000)
  OFFSET = TH / 2 (32'h20000000)
  NOISE  = 32-bit Uniform (LFSR Galois)
  SEED   = مختلف لكل NCO (لضمان تنوع الإطلاقات)
```

### 3.2 المعاملات (v2.1.0)

| المعامل | القيمة | الوصف |
|---------|--------|-------|
| `N_OSC` | 6 | عدد المذبذبات في الخزان |
| `N_WINDOW` | **5** | عدد دورات الساعة لكل تصنيف (v2.1.0) |
| `ACC_WIDTH` | 32 | عرض الـ accumulator (signed) |
| `TH` | 2^30 | عتبة الإطلاق |
| `OFFSET` | TH/2 | مقدار إعادة الضبط |
| `NOISE` | Uniform (32-bit) | ضوضاء رقمية من LFSR مدمج |

### 3.3 آلية التصويت (FDE)

```
لكل عينة:
  1. تشغيل 6 NCOs لمدة 5 دورات ساعة ← كل NCO يحسب firing_dir + fire_count
  2. pos_votes = عدد NCOs حيث firing_dir = 1
  3. neg_votes = عدد NCOs حيث firing_dir = 0
  4. إذا pos_votes > neg_votes → القرار = موجب (+1)
  5. إذا neg_votes > pos_votes → القرار = سالب (-1)
  6. إذا تعادل → مجموع fire_counts يفك التعادل
```

### 3.4 اختلافات عن Python prototype

| الخاصية | Python (تجريبي) | Verilog RTL (v2.1.0) |
|---------|----------------|----------------------|
| N_STEPS / N_WINDOW | 2 | **5** |
| الضوضاء | Gaussian (random.gauss) | **Uniform (LFSR 32-bit)** |
| seeds | random.Random(seed) لكل NCO | LFSR ثابت مدمج لكل NCO |
| الطاقة | 0.03 pJ (تقدير جمع فقط) | **~6 pJ (من LUTs + FFs حقيقي)** |

## 📋 القسم 6: القيود المعروفة

1. **Uniform noise بدلاً من Gaussian:** LFSR يولد ضوضاء منتظمة التوزيع. هذا يعمل عملياً (100% دقة) لكنه لا يمثل نموذج Stochastic Resonance الكلاسيكي. يمكن تحسينه لاحقاً باستخدام LUT Gaussian (Box-Muller) على حساب المساحة.

2. **N_WINDOW=5 (وليس 2):** Python حققت 100% عند N_STEPS=2، لكن Verilog الحالي يحتاج 5 دورات لضمان 100% مع LFSR الحالي. تحسين LFSR إلى مدى أوسع قد يسمح بـ N_WINDOW=2.

3. **1,455 LUTs > تقدير 500 LUT السابق:** التقدير الأولي كان متفائلاً. 1,455 LUT تبقى 27% فقط من iCE40UP5K — مقبول جداً.

4. **لا يوجد P&R (nextpnr):** Yosys synthesis فقط. التوقيت الدقيق والطاقة الحقيقية يحتاجان P&R فِعلي. هذا معلّق لحين توفر بيئة Linux أو FPGA حقيقي.
